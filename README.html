PsychoPy Plugin + ESP32 Firmware

psychopy-apparatus

A combined software and firmware stack for the Apparatus: a custom setup for measuring force, driving actuators, reading pegboard state, and exposing all of it to PsychoPy experiments on the PC.

2ESP32 nodes: server + client
250 SPSADS1115 force backend target rate
21logical pegboard holes
ESP-NOWpaired wireless link between nodes

Repository Layout

  • psychopy_apparatus/: PsychoPy plugin package
  • firmware/esp_server/: server ESP32 firmware
  • firmware/esp_client/: client ESP32 firmware
  • firmware/tests/: serial, force, LED, and reed utilities
  • experiments/: notes and application documentation

Installation

  1. Install PsychoPy 2025.2.1 or later.
  2. If the plugin manager is broken, open 2025.1.1 once before upgrading.
  3. Clone this repository.
  4. In Builder, open Plugins & Packages Manager.
  5. Install from pyproject.toml using the Python project filter.

Server Role

The server is the PC-facing bridge. It handles local actuation and force sensing, then relays client traffic over ESP-NOW.

  • CMD_MOTOR (0x01)
  • CMD_LIGHT (0x02)
  • CMD_FORCE_START (0x03)
  • CMD_FORCE_STOP (0x04)
  • CMD_MAGNET (0x05)
  • DATA_FORCE (0x90) as time_us + value(0.1N) + device
Default force acquisition uses an ADS1115 over I2C on GPIO 25/26, scanning addresses 0x48 through 0x4B.

Client Role

The client is the pegboard node. It controls LEDs, streams reed states, and performs hole-specific hall measurements.

  • CMD_LED_SET_N (0x10)
  • CMD_LED_SHOW (0x11)
  • CMD_HOLE_START (0x12)
  • CMD_HOLE_STOP (0x13)
  • CMD_REED_START (0x14)
  • CMD_REED_STOP (0x15)
  • DATA_REED (0x91)
  • DATA_HALL (0x92)

Pairing and Data Flow

The PC talks to the server over USB serial. The server and client talk over ESP-NOW on fixed channel 1. Pairing state is stored in ESP32 NVS, so it usually survives reboot and normal reflashing.

  1. Unpaired devices broadcast HELLO every 500 ms.
  2. The server sends PAIR_OFFER.
  3. The client replies with PAIR_ACCEPT.
  4. The server sends PAIR_CONFIRM and both sides persist the pairing.

If a replaced device refuses normal traffic, erase NVS or fully erase flash before re-pairing.

Force Pipeline

  1. Read filtered ADS1115 millivolts.
  2. Subtract tare baseline.
  3. Convert to volts.
  4. Apply calibration: 176.8325 N/V, intercept -19.5057 N.
  5. Encode as signed int16 in 0.1 N units.

Tare uses 12 warmup cycles, 96 accumulation cycles, and a 1.5 s timeout.

Debug Priorities

  • No force stream: verify ADS1115 wiring, power, and address first.
  • Bad force values: inspect tare stability and calibration assumptions.
  • Discovery works but commands do not: check for stale pairing info in NVS.
  • Force works but client data does not: inspect the client I2C path and devices.

Quick Orientation

The repo splits cleanly into two halves: PC-side PsychoPy integration in psychopy_apparatus/ and embedded code in firmware/. If you need the shortest route into the codebase, start with the overview doc in firmware/EXPERIMENTAL_CLIENT_SERVER_OVERVIEW.md, then open the matching sketch or package.

Use this tab as a fast map: where code lives, which file is the entry point, and which scripts are meant for testing rather than production firmware.

Firmware Entry Points

firmware/esp_server/esp_server.ino

Main server firmware. This is the file for force streaming, motor control, magnets, light input, serial framing, and server-side pairing behavior.

firmware/esp_client/esp_client.ino

Main client firmware. This is the file for peg LEDs, reed readout, hall sensing, and hole-selection behavior.

firmware/esp_server_experimental/esp_server_experimental.ino

Alternate server branch for experiments that should not immediately replace the main server sketch.

firmware/EXPERIMENTAL_CLIENT_SERVER_OVERVIEW.md

Fast architecture document covering protocol, pairing, ADS1115 behavior, client responsibilities, and common debug paths.

Rule of Thumb

  • If the issue is PsychoPy setup or packaging, start in psychopy_apparatus/ and pyproject.toml.
  • If the issue is force, pairing, or server-side command handling, start in firmware/esp_server/esp_server.ino.
  • If the issue is peg LEDs, reed switches, or hall sensing, start in firmware/esp_client/esp_client.ino.
  • If you need context before opening code, read firmware/EXPERIMENTAL_CLIENT_SERVER_OVERVIEW.md first.
